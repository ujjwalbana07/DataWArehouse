5.5 Data Transformation and Cleansing Rules 
Rule T1: OK Flag Filtering 
SSIS Component: Conditional Split (in Load stg_Movement Data Flow) Condition: OK_FLAG == 1 && 
MOVE > 0 && PRICE > 0 Effect: Rows with corrupt data (OK = 0), zero/negative movement, or 
zero/negative price are routed to the rejected output and counted. 
Rule T2: SALE Code Standardization 
SSIS Component: Derived Column Expression: ISNULL(SALE) || TRIM(SALE) == "" ? "NONE" : 
UPPER(TRIM(SALE)) Result: Blank/null → 'NONE'. All codes uppercased and trimmed. Valid values: B, 
C, S, NONE. 
Rule T3: QTY Null/Zero Handling 
SSIS Component: Derived Column Expression: ISNULL(QTY) || QTY <= 0 ? 1 : QTY Rationale: QTY 
is the divisor in dollar_sales. Default 1 prevents division-by-zero. 
Rule T4: PROFIT Null Handling 
SSIS Component: Derived Column Expression: ISNULL(PROFIT) ? 0.00 : PROFIT 
Rule T5: DESCRIP Special Character Cleansing 
SSIS Component: Derived Column (in Load stg_UPC Data Flow) Expression: 
TRIM(REPLACE(REPLACE(REPLACE(REPLACE(REPLACE(DESCRIP,"#",""),"~",""),"<",""),"$",""),"
*","")) Null check: ISNULL(result) || result == "" ? "Unknown" : result 
Rule T6: INCOME Log-to-Dollar Conversion 
SSIS Component: OLE DB Source SQL query (in Load Dim_Store Data Flow) SQL: SELECT ..., 
EXP(LOG_INCOME) AS MEDIAN_INCOME_DOLLARS, ... FROM stg_Demo Rationale: INCOME = 
ln(median income). EXP() recovers dollar value. Example: EXP(10.9) ≈ $54,176. 
Rule T7: Income Band Derivation 
SSIS Component: Derived Column (after T6) Expression: MEDIAN_INCOME_DOLLARS >= 60000 ? 
"High" : (MEDIAN_INCOME_DOLLARS >= 35000 ? "Medium" : "Low") 
Rule T8: Dollar Sales Computation 
SSIS Component: Derived Column (in Load Fact_Weekly_Sales Data Flow) Expression: 
(DT_DECIMAL,2)((PRICE * MOVE) / QTY) Rationale: Standard DFF revenue formula. QTY = pricing 
bundle quantity. 
Rule T9: Dim_Date Calendar Derivation 
SSIS Component: Execute SQL Task Logic: DISTINCT WEEK_IDs from stg_Movement UNION 
stg_Ccount. For each: calendar_date = DATEADD(WEEK, WEEK_ID − 1, '1989-09-14'). Derive month, 
quarter, year, season, holiday_flag. 
Rule T10: Surrogate Key Lookup with In-Pipeline Unknown Member Replacement 
SSIS Component: Lookup Transformation + Derived Column (per dimension) Configuration: Each 
Lookup is set to "Redirect rows to no match output." Unmatched rows flow to a Derived Column that 
replaces the NULL surrogate key with −1. A Union All then merges matched and unmatched rows before 
the OLE DB Destination. 
CRITICAL DESIGN NOTE: The fact table FK columns are defined as NOT NULL. Therefore, NULL 
values from failed lookups cannot reach the OLE DB Destination — they must be replaced with −1 inside 
the SSIS pipeline before the INSERT occurs. The Derived Column expressions are: 
store_sk_final    = ISNULL(store_sk) ? -1 : store_sk 
date_sk_final     = ISNULL(date_sk) ? -1 : date_sk 
product_sk_final  = ISNULL(product_sk) ? -1 : product_sk 
promotion_sk_final = ISNULL(promotion_sk) ? -1 : promotion_sk 
  
Rule T11: Customer Count Weekly Aggregation 
SSIS Component: OLE DB Source with GROUP BY SQL: SELECT STORE_ID, WEEK_ID, 
SUM(CUSTOMER_COUNT) AS CUSTOMER_COUNT FROM stg_Ccount GROUP BY STORE_ID, 
WEEK_ID Rationale: Ccount may contain daily rows. Fact grain is Store × Week, so daily → weekly 
aggregation. 
5.6 Plan for Aggregate Tables 
Three aggregate tables (Materialized Indexed Views in SQL Server) pre-compute common roll-ups: 
AGG_Monthly_Store_Revenue (BQ1 dashboards): 
CREATE VIEW dbo.AGG_Monthly_Store_Revenue WITH SCHEMABINDING AS 
SELECT f.store_sk, d.calendar_year, d.calendar_month, 
    SUM(f.dollar_sales) AS total_revenue, 
    SUM(f.units_sold) AS total_units, 
    COUNT_BIG(*) AS row_count 
FROM dbo.Fact_Weekly_Sales f 
INNER JOIN dbo.Dim_Date d ON f.date_sk = d.date_sk 
GROUP BY f.store_sk, d.calendar_year, d.calendar_month; 
GO 
CREATE UNIQUE CLUSTERED INDEX IX_AGG_Monthly 
ON dbo.AGG_Monthly_Store_Revenue(store_sk, calendar_year, calendar_month); 
  
AGG_Weekly_Store_Sales (BQ5 drill-across pre-aggregation): 
CREATE VIEW dbo.AGG_Weekly_Store_Sales WITH SCHEMABINDING AS 
SELECT f.store_sk, f.date_sk, 
    SUM(f.dollar_sales) AS total_revenue, 
    SUM(f.units_sold) AS total_units, 
    COUNT_BIG(*) AS row_count 
FROM dbo.Fact_Weekly_Sales f 
GROUP BY f.store_sk, f.date_sk; 
GO 
CREATE UNIQUE CLUSTERED INDEX IX_AGG_Weekly 
ON dbo.AGG_Weekly_Store_Sales(store_sk, date_sk); 
  
AGG_Promo_Performance (BQ3 promotion analysis): 
CREATE VIEW dbo.AGG_Promo_Performance WITH SCHEMABINDING AS 
SELECT f.promotion_sk, 
    SUM(f.dollar_sales) AS total_revenue, 
    SUM(f.units_sold) AS total_units, 
    COUNT_BIG(*) AS row_count 
FROM dbo.Fact_Weekly_Sales f 
GROUP BY f.promotion_sk; 
GO 
CREATE UNIQUE CLUSTERED INDEX IX_AGG_Promo 
ON dbo.AGG_Promo_Performance(promotion_sk);

Staging Principles: 
1. Staging tables TRUNCATED before each ETL run (idempotent). 
2. Source-level cleansing (OK filter, nulls, type conversion) during source → staging. 
3. Business transformations (EXP, income_band, dollar_sales, SK lookups) during staging → DW. 
4. All staging tables DROPPED after DW loading is validated.

5.9.4 Dim_Date (Execute SQL Task) 
DECLARE @epoch DATE = '1989-09-14'; 
 
INSERT INTO Dim_Date (week_id, calendar_date, calendar_month, calendar_quarter, 
                      calendar_year, season, holiday_flag) 
SELECT DISTINCT 
    w.WEEK_ID, 
    DATEADD(WEEK, w.WEEK_ID - 1, @epoch), 
    MONTH(DATEADD(WEEK, w.WEEK_ID - 1, @epoch)), 
    DATEPART(QUARTER, DATEADD(WEEK, w.WEEK_ID - 1, @epoch)), 
    YEAR(DATEADD(WEEK, w.WEEK_ID - 1, @epoch)), 
    CASE 
        WHEN MONTH(DATEADD(WEEK, w.WEEK_ID - 1, @epoch)) IN (3,4,5) THEN 'Spring' 
        WHEN MONTH(DATEADD(WEEK, w.WEEK_ID - 1, @epoch)) IN (6,7,8) THEN 'Summer' 
        WHEN MONTH(DATEADD(WEEK, w.WEEK_ID - 1, @epoch)) IN (9,10,11) THEN 'Fall' 
        ELSE 'Winter' 
    END, 
    CASE 
        WHEN MONTH(DATEADD(WEEK, w.WEEK_ID - 1, @epoch)) = 11 
             AND DAY(DATEADD(WEEK, w.WEEK_ID - 1, @epoch)) BETWEEN 22 AND 28 THEN 1 
        WHEN MONTH(DATEADD(WEEK, w.WEEK_ID - 1, @epoch)) = 12 
             AND DAY(DATEADD(WEEK, w.WEEK_ID - 1, @epoch)) BETWEEN 20 AND 31 THEN 1 
        ELSE 0 
    END 
FROM ( 
    SELECT WEEK_ID FROM [Ujjwal_Group3_Staging_Area].dbo.stg_Movement 
    UNION 
    SELECT WEEK_ID FROM [Ujjwal_Group3_Staging_Area].dbo.stg_Ccount 
) w 
WHERE w.WEEK_ID IS NOT NULL AND w.WEEK_ID > 0 
ORDER BY w.WEEK_ID; 
  
5.10 ETL for Fact Tables 
5.10.1 Fact_Weekly_Sales (Data Flow Task) — CORRECTED DESIGN 
CRITICAL FIX: The fact table FK columns are NOT NULL. The Lookup Transformations for unmatched 
rows produce NULL surrogate keys. If NULLs reach the OLE DB Destination, the INSERT will fail and 
crash the SSIS package. Therefore, a Derived Column must replace NULLs with −1 inside the pipeline 
before the data reaches the destination. 
SSIS Data Flow (corrected execution order): 
┌────────────────────────────────┐ 
│  OLE DB Source                 │  SELECT STORE_ID, UPC_ID, WEEK_ID, 
│  (Ujjwal_Group3_Staging_Area)  │  MOVE, QTY, PRICE, SALE_CODE, PROFIT 
│                                │  FROM stg_Movement 
└──────────────┬─────────────────┘ 
               ↓ 
┌────────────────────────────────┐ 
│  Lookup #1: Store SK           │  Match: STORE_ID = Dim_Store.store_id 
│  No match → Redirect           │  Output: store_sk (NULL if no match) 
└──────────────┬─────────────────┘ 
               ↓ 
┌────────────────────────────────┐ 
│  Lookup #2: Date SK            │  Match: WEEK_ID = Dim_Date.week_id 
│  No match → Redirect           │  Output: date_sk (NULL if no match) 
└──────────────┬─────────────────┘ 
               ↓ 
┌────────────────────────────────┐ 
│  Lookup #3: Product SK         │  Match: UPC_ID = Dim_Product.upc_id 
│  No match → Redirect           │  Output: product_sk (NULL if no match) 
└──────────────┬─────────────────┘ 
               ↓ 
┌────────────────────────────────┐ 
│  Lookup #4: Promotion SK       │  Match: SALE_CODE = Dim_Promotion.sale_code 
│  No match → Redirect           │  Output: promotion_sk (NULL if no match) 
└──────────────┬─────────────────┘ 
               ↓ 
┌────────────────────────────────────────────────┐ 
│  Derived Column: "Replace NULL SKs with −1"    │  ← CRITICAL FIX 
│                                                 │ 
│  store_sk_safe = ISNULL(store_sk) ? -1 : store_sk 
│  date_sk_safe = ISNULL(date_sk) ? -1 : date_sk 
│  product_sk_safe = ISNULL(product_sk) ? -1 : product_sk 
│  promotion_sk_safe = ISNULL(promotion_sk) ? -1 : promotion_sk 
└──────────────┬──────────────────────────────────┘ 
               ↓ 
┌────────────────────────────────────────────────┐ 
│  Derived Column: "Compute dollar_sales"         │ 
│                                                 │ 
│  dollar_sales = (DT_DECIMAL,2)((PRICE * MOVE) / QTY) 
└──────────────┬──────────────────────────────────┘ 
               ↓ 
┌────────────────────────────────┐ 
│  OLE DB Destination            │  Table: Fact_Weekly_Sales 
│  Column Mappings:              │ 
│    store_sk_safe → store_sk    │  (guaranteed NOT NULL: −1 or valid SK) 
│    date_sk_safe → date_sk      │ 
│    product_sk_safe → product_sk│ 
│    promotion_sk_safe → promotion_sk 
│    MOVE → units_sold           │ 
│    PRICE → retail_price        │ 
│    dollar_sales → dollar_sales │ 
│    PROFIT → profit_margin_pct  │ 
└────────────────────────────────┘ 
  
Why this design is safe: Every row reaching the OLE DB Destination is guaranteed to have non-NULL 
integer values in all four FK columns. Valid lookups produce the correct surrogate key; failed lookups 
produce −1 (Unknown Member). The NOT NULL constraints are always satisfied. No post-load UPDATE 
script is needed.

6.6 Data Granularity at the Independent Data Mart Level 
Fact_Weekly_Sales: One row per Store × Product (UPC) × Week — the finest grain from the Movement 
source. Each row captures the complete weekly sales record for one product at one store. 
Fact_Customer_Traffic: One row per Store × Week — the only grain from CCOUNT data, which has no 
product-level breakdown. 
Why separate tables are mandatory: If customer_count (Store × Week) were embedded into the sales fact 
(Store × Product × Week), each store's weekly count would replicate across every product row. Example: 
Store #5 has 10,000 visitors and 5,000 products in Week 278. Merged table → SUM(customer_count) = 
50,000,000 instead of 10,000. BQ5 uses Kimball drill-across (pre-aggregate sales to Store × Week, then join 
to traffic) to avoid this.

BQ5: Customer Count vs. Sales (Drill-Across) 
WITH SalesAgg AS ( 
    SELECT f.store_sk, f.date_sk, 
        SUM(f.dollar_sales) AS total_revenue, SUM(f.units_sold) AS total_units 
    FROM Fact_Weekly_Sales f GROUP BY f.store_sk, f.date_sk 
), 
TrafficData AS ( 
    SELECT t.store_sk, t.date_sk, t.customer_count 
    FROM Fact_Customer_Traffic t 
) 
SELECT s.store_id, s.store_zone, s.income_band, d.week_id, d.calendar_year, 
    sa.total_revenue, sa.total_units, td.customer_count, 
    CASE WHEN td.customer_count > 0 
         THEN CAST(sa.total_revenue / td.customer_count AS DECIMAL(10,2)) ELSE 0 
    END AS revenue_per_visitor 
FROM SalesAgg sa 
JOIN TrafficData td ON sa.store_sk = td.store_sk AND sa.date_sk = td.date_sk 
JOIN Dim_Store s ON sa.store_sk = s.store_sk 
JOIN Dim_Date d ON sa.date_sk = d.date_sk 
WHERE s.store_sk != -1 AND d.date_sk != -1 
ORDER BY revenue_per_visitor DESC; 
  
Why this works: SalesAgg CTE rolls up from Store × Product × Week to Store × Week first (eliminating 
product dimension via SUM). The join to TrafficData is then 1:1 at Store × Week grain. No fan-out. No 
double-counting. This is the Kimball drill-across pattern.

6.11 Special ETL Techniques Implemented 
Technique 1: Log-Income to Dollar Conversion (EXP Function) 
DEMO.csv stores INCOME as ln(median household income). During the Dim_Store load, the OLE DB 
Source query applies EXP(LOG_INCOME) to recover dollar values. Example: INCOME = 10.9 → 
EXP(10.9) ≈ $54,176. This feeds the income_band Derived Column (High ≥ $60K, Medium ≥ $35K, Low < 
$35K). Without this transformation, the income_band thresholds would be applied to logarithmic values, 
producing meaningless results. 
Technique 2: QTY-Based Revenue Normalization 
The dollar_sales formula — (PRICE × MOVE) / QTY — handles multi-buy pricing bundles. A 2-for-$5 
deal has QTY = 2, PRICE = $5; without dividing by QTY, revenue would be overstated for every bundled 
product. The QTY default of 1 (Rule T3) prevents division-by-zero. 
Technique 3: In-Pipeline Unknown Member Replacement (NOT NULL Safety) 
This is the most critical SSIS technique in the project. The fact table FK columns are NOT NULL, but 
Lookup Transforms produce NULL for unmatched rows. If NULLs reach the OLE DB Destination, the 
INSERT fails and the package crashes. The solution: a Derived Column immediately after all Lookups 
replaces every NULL SK with −1 using ISNULL(sk) ? -1 : sk. This guarantees every row entering the 
destination has valid, non-NULL FK values — either the correct surrogate key or the Unknown Member 
(−1). 
Technique 4: Kimball Drill-Across for BQ5 
BQ5 requires combining two fact tables at different grains. The CTE-based query first aggregates 
Fact_Weekly_Sales from Store × Product × Week down to Store × Week (eliminating the product 
dimension), then joins to Fact_Customer_Traffic on the conformed dimension keys. Revenue-per-visitor is 
computed as a derived metric. This avoids the grain-violation error that would occur with a direct join. 
Technique 5: Conditional Split with Row Count Auditing 
The stg_Movement Data Flow uses a Conditional Split to enforce OK = 1, MOVE > 0, and PRICE > 0. 
Rejected rows flow to a Row Count transformation that stores the count in SSIS variable @RejectedCount. 
This provides full audit transparency: we know exactly how many source rows were excluded and why.

8. Conclusion 
This integrated report presents the complete data warehouse lifecycle for Dominick's Finer Foods — from 
requirements gathering and domain research (Sections 1–2), through Kimball dimensional modeling and star 
schema design (Sections 3–4), to ETL planning and SSIS implementation with full evidence (Sections 5–6). 
The central design decision — two separate fact tables with grain isolation — is mandatory because 
checkout transactions (Store × Product × Week) and customer traffic (Store × Week) are incompatible 
grains. Merging them would inflate traffic counts by the product count per store-week, corrupting BQ5. 
Conformed Dim_Store and Dim_Date enable safe drill-across analysis. 
The ETL pipeline, implemented entirely in SSIS on SQL Server 2016, includes 11 transformation rules: OK 
filtering, QTY normalization, SALE standardization, DESCRIP cleansing, PROFIT handling, 
EXP(INCOME) conversion, income_band derivation, dollar_sales computation, calendar derivation, in-
pipeline NULL-to-Unknown-Member SK replacement, and customer count weekly aggregation. All five 
business questions return correct, aggregation-safe results. Staging tables have been dropped per the 
documented lifecycle.

10. Appendix: Assumptions and Technical Notes 
A1. Dollar sales = (PRICE × MOVE) / QTY. QTY = pricing bundle quantity (2-for-$5 → QTY = 2). 
A2. WEEK values in CCOUNT.csv and Movement files share the same numbering system. 
A3. Dim_Date epoch = September 14, 1989 (Week 1). Adjustable without schema changes. 
A4. SCD Type 1 (overwrite) for demographics — static 1990 Census values. 
A5. INCOME = ln(median income). EXP() recovers dollars. EXP(10.9) ≈ $54,176. 
A6. profit_margin_pct stored as cents-per-dollar (25.30 = 25.30%). Use AVG(), not SUM(). 
A7. OK = 0 rows permanently excluded. MOVE ≤ 0 and PRICE ≤ 0 also rejected. 
A8. Unknown Members at SK = −1 in all dimensions. NULL SKs replaced with −1 inside SSIS pipeline 
(Derived Column) before INSERT, respecting NOT NULL constraints. 
A9. All ETL performed exclusively in SSIS. No Python/Java/Excel for transformations. 
A10. Databases: Staging = Ujjwal_Group3_Staging_Area, DW = Ujjwal_Group3_DW_Area.
